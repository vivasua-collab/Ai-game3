## 1. Схема A: Текущая (чекпоинты + worklog)

### 1.1. Артефакты

| Артефакт | Объём | ≈ Токенов | Когда читается | Назначение |
|----------|-------|-----------|----------------|------------|
| `START_PROMPT.md` | 7 KB | ~2 400 | Каждый старт | Правила работы, структура папок, запреты |
| `Caveman.md` | 4 KB | ~1 400 | Каждый старт | Режим коммуникации |
| `worklog.md` | 102 KB | ~34 000 | Каждый старт (хвост) | Хроника работы всех агентов |
| `checkpoints/` (116 файлов) | 642 KB | ~214 000 | Выборочно | Фиксация этапов работы |

### 1.2. Рабочий процесс

```
Старт сессии:
  1. Прочитать START_PROMPT.md         → ~2 400 токенов
  2. Прочитать Caveman.md              → ~1 400 токенов
  3. Прочитать worklog.md (хвост)      → ~3 000-5 000 токенов
  4. Прочитать последние чекпоинты     → ~2 000-5 000 токенов
  ─────────────────────────────────────
  Итого при старте:                    ~8 000-14 000 токенов

Во время работы:
  - Обновить worklog.md (append)
  - Создать чекпоинт checkpoints/ММ_ДД_цель.md
  - Git push при завершении

Следующая сессия:
  - Контекст из предыдущей сессии ПОТЕРЯН
  - Всё восстанавливается через чтение файлов
  - Чекпоинты неструктурированы — агент не знает, какие релевантны
```

### 1.3. Проблемы

| # | Проблема | Влияние |
|---|---------|---------|
| P1 | **Дорогостоящий старт** — чтение файлов каждый раз | 8-14K токенов тратится до начала работы |
| P2 | **Нет выборочного доступа** — всё или ничего | Агент читает нерелевантные чекпоинты или не читает нужные |
| P3 | **Нет поиска** — только чтение по имени файла | Невозможно найти «решение про орбитальное оружие» без знания имени файла |
| P4 | **Дублирование** — чекпоинты повторяют друг друга | 116 файлов, многие перекрываются по содержимому |
| P5 | **Нет связей** — факты изолированы | Решение в чекпоинте #47 не связано с фактом в #23 |
| P6 | **Нет забывания** — всё копится | 642 KB чекпоинтов, 102 KB worklog, и растёт |
| P7 | **worklog бесполезен после N сессий** | 34K токенов текста, найти что-то — невозможно |
| P8 | **Ручное создание чекпоинтов** | Агент забывает — чекпоинт не создан, контекст потерян |

## 2. Схема B: Полная замена (SQLite + JSONL + L1/L2/L3)

### 2.1. Три уровня (кэш-модель)

```
L1 — MEMORY.md           → Всегда в контексте, <500 токенов
                             Автогенерация при каждом «сне»
                             ЗАМЕНА чтения чекпоинтов при старте

L2 — SQLite + FTS5       → По запросу, когда нужна конкретика
                             python3 memory/query.py "запрос"
                             ~200 токенов на запрос

L3 — JSONL логи          → Source of truth, append-only
                             Один файл на день
                             Git-мержимый (текстовый)
                             SQLite — производное, пересоздаётся
```

### 2.2. Структура файлов

```
/home/z/my-project/
├── MEMORY.md                    # L1 — автогенерация, <500 токенов
├── memory/
│   ├── log/                     # L3 — append-only JSONL (Git-мержимые)
│   │   ├── 2026-04-20.jsonl
│   │   ├── 2026-04-21.jsonl
│   │   └── 2026-04-22.jsonl
│   ├── db/                      # L2 — SQLite (gitignored, производное)
│   │   └── agent_memory.db      # Генерируется из JSONL при старте
│   ├── query.py                 # CLI: поиск по памяти
│   ├── log.py                   # CLI: добавление записи
│   ├── sleep.py                 # CLI: цикл «сна»
│   └── schema.sql               # Схема БД + миграции
├── START_PROMPT.md              # Остаётся (правила работы)
├── Caveman.md                   # Остаётся (режим коммуникации)
└── ...
```

### 2.3. Схема SQLite

```sql
-- Эпизоды (контекст: когда, почему был сделан вывод)
CREATE TABLE episodes (
    id            INTEGER PRIMARY KEY,
    session_id    TEXT NOT NULL,
    date          TEXT NOT NULL,
    trigger_type  TEXT NOT NULL,        -- user_request | auto | sleep
    summary       TEXT,
    created_at    TEXT NOT NULL
);

-- Факты (атомарные утверждения о проекте)
CREATE TABLE facts (
    id              INTEGER PRIMARY KEY,
    content         TEXT NOT NULL,
    type            TEXT NOT NULL,      -- knowledge | constraint | dependency | warning
    confidence      REAL DEFAULT 0.8,
    valid_from      TEXT NOT NULL,
    valid_until     TEXT,               -- NULL = актуален
    superseded_by   INTEGER REFERENCES facts(id),
    source_episode  INTEGER REFERENCES episodes(id),
    access_count    INTEGER DEFAULT 0,
    last_accessed   TEXT,
    is_active       INTEGER DEFAULT 1,
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL
);

-- Решения (архитектурные/технические выборы)
CREATE TABLE decisions (
    id            INTEGER PRIMARY KEY,
    what          TEXT NOT NULL,
    why           TEXT NOT NULL,
    alternatives  TEXT,
    is_frozen     INTEGER DEFAULT 0,
    source_episode INTEGER REFERENCES episodes(id),
    created_at    TEXT NOT NULL
);

-- Задачи
CREATE TABLE tasks (
    id            INTEGER PRIMARY KEY,
    title         TEXT NOT NULL,
    description   TEXT,
    status        TEXT NOT NULL,        -- active | completed | blocked | archived
    parent_task   INTEGER REFERENCES tasks(id),
    source_episode INTEGER REFERENCES episodes(id),
    created_at    TEXT NOT NULL,
    updated_at    TEXT NOT NULL
);

-- Связи (граф знаний)
CREATE TABLE entity_links (
    source_type   TEXT NOT NULL,
    source_id     INTEGER NOT NULL,
    target_type   TEXT NOT NULL,
    target_id     INTEGER NOT NULL,
    relation      TEXT NOT NULL,        -- depends_on | justifies | contradicts | supersedes | blocks
    created_at    TEXT NOT NULL,
    PRIMARY KEY (source_type, source_id, target_type, target_id, relation)
);

-- FTS5 полнотекстовый поиск
CREATE VIRTUAL TABLE facts_fts USING fts5(content, type, content=facts, content_rowid=id);
CREATE VIRTUAL TABLE decisions_fts USING fts5(what, why, content=decisions, content_rowid=id);
CREATE VIRTUAL TABLE tasks_fts USING fts5(title, description, content=tasks, content_rowid=id);

-- Триггеры FTS5 (пример для facts, аналогично для decisions/tasks)
CREATE TRIGGER facts_ai AFTER INSERT ON facts BEGIN
    INSERT INTO facts_fts(rowid, content, type) VALUES (new.id, new.content, new.type);
END;
CREATE TRIGGER facts_ad AFTER DELETE ON facts BEGIN
    INSERT INTO facts_fts(facts_fts, rowid, content, type) VALUES('delete', old.id, old.content, old.type);
END;
CREATE TRIGGER facts_au AFTER UPDATE ON facts BEGIN
    INSERT INTO facts_fts(facts_fts, rowid, content, type) VALUES('delete', old.id, old.content, old.type);
    INSERT INTO facts_fts(rowid, content, type) VALUES (new.id, new.content, new.type);
END;

-- Метаданные
CREATE TABLE metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

### 2.6. Цикл «сна»

```
Light sleep (без LLM, мгновенный)          Deep sleep (с LLM, раз в день)
───────────────────────────────          ───────────────────────────────
• JSONL → SQLite импорт                  • LLM-извлечение фактов/решений
• FTS5 обновление индексов               • Дедупликация по FTS5
• Пометка обработанных логов             • Decay (устаревшие → is_active=false)
                                          • Генерация MEMORY.md
                                          • Git add + commit + push
```

### 2.7. Рабочий процесс (схема B)

```
Старт: MEMORY.md + START_PROMPT.md + Caveman.md = ~4 000 токенов
Работа: query.py для поиска, log.py для записи
Конец:  sleep.py --deep → обновление MEMORY.md → git push
```

## 3. Схема C: Чекпоинты + результирующий файл (компромисс)

### 3.0. Идея

Текущую схему НЕ ломаем. Чекпоинты остаются как есть. Добавляем **один автогенерируемый файл** — компактную выжимку последних сессий, который читается при старте вместо сканирования чекпоинтов и worklog.

```
Старт сейчас:     START_PROMPT + Caveman + worklog + чекпоинты = 8-14K
Старт схема C:    START_PROMPT + Caveman + SESSION_SUMMARY.md  = ~4.2K
```

### 3.1. Структура файлов

```
/home/z/my-project/
├── SESSION_SUMMARY.md           # НОВЫЙ — автогенерация, <500 токенов
├── checkpoints/                 # БЕЗ ИЗМЕНЕНИЙ
├── worklog.md                   # БЕЗ ИЗМЕНЕНИЙ
├── START_PROMPT.md              # БЕЗ ИЗМЕНЕНИЙ
├── Caveman.md                   # БЕЗ ИЗМЕНЕНИЙ
└── ...
```

Никаких новых директорий. Никаких зависимостей. Один файл.


### 3.3. Механизм обновления

| Вариант | Как | Зависимости | Качество |
|---------|-----|-------------|----------|
| **C1. Ручной** | Агент пишет/обновляет SESSION_SUMMARY.md сам в конце сессии | Нет | Высокое (агент знает что делал) |
| **C2. Скрипт из чекпоинтов** | `python3 gen_summary.py` читает последние чекпоинты | Python | Среднее (без LLM — только заголовки/даты) |
| **C3. LLM-суммаризация** | `python3 gen_summary.py --llm` | Python + LLM API | Высокое |

**Рекомендация:** Начать с **C1** (ноль зависимостей).

### 3.4. Правила для SESSION_SUMMARY.md

1. **Глубина:** последние 5 дней активности (не календарных)
2. **Лимит:** <500 токенов (~1.5 KB)
3. **Обновление:** при завершении каждой сессии
4. **Включать:** ключевые действия за день (2-5 пунктов), активные задачи, замороженные решения, предупреждения
5. **НЕ включать:** детали реализации, историю багов, подробности диалогов

### 3.5. Интеграция в START_PROMPT.md

Добавить одну строку:

```markdown
**Контекст последних сессий:** `SESSION_SUMMARY.md` — читать при старте вместо чекпоинтов.
```

### 3.6. Рабочий процесс (схема C)

```
Старт сессии:
  1. Прочитать START_PROMPT.md         → ~2 400 токенов
  2. Прочитать Caveman.md              → ~1 400 токенов
  3. Прочитать SESSION_SUMMARY.md      → ~400 токенов
  ─────────────────────────────────────
  Итого:                               ~4 200 токенов

Во время работы:
  - Всё как раньше: чекпоинты, worklog, git push
  - При необходимости — читать конкретный чекпоинт

Конец сессии:
  - Обновить SESSION_SUMMARY.md (вручную / скрипт)
  - Создать чекпоинт
  - git add + commit + push
```

## 4. Сравнение трёх схем

### 4.1. Карточки схем

| Критерий | A: Текущая | B: Полная замена | C: Чекпоинты + саммари |
|----------|-----------|-----------------|----------------------|
| **Токены при старте** | 8-14K | ~4K | ~4.2K |
| **Экономия vs текущая** | — | 50-70% | 50-70% |
| **Поиск по памяти** | ❌ | ✅ FTS5 + векторы | ❌ |
| **Структура знаний** | ❌ | ✅ факты/решения/связи | ❌ (саммари только) |
| **Автоконсолидация** | ❌ | ✅ цикл «сна» | ❌ |
| **Забывание/decay** | ❌ | ✅ | ❌ |
| **Сложность внедрения** | — | Высокая | **Низкая** |
| **Новые зависимости** | — | Python + SQLite + LLM | **Нет** (вариант C1) |
| **Риск поломки** | — | Средний | **Нулевой** |
| **Обратная совместимость** | — | Частичная | **Полная** |
| **Масштабируемость** | ❌ | ✅ | ⚠️ (до ~50 чекпоинтов ок) |

### 4.2. Что получаем при переходе A → C

| # | Получаем | Оценка |
|---|---------|--------|
| 1 | Экономия 50-70% токенов при старте | ⭐⭐⭐ |
| 2 | Нулевой риск, текущий процесс не ломается | ⭐⭐⭐ |
| 3 | Полная обратная совместимость | ⭐⭐⭐ |
| 4 | Человекочитаемый актуальный срез | ⭐⭐ |
| 5 | Можно внедрить за 10 минут | ⭐⭐⭐ |

### 4.3. Что НЕ получаем (C vs B)

| # | Ограничение | Значимость |
|---|-----------|------------|
| 1 | Поиск по памяти (FTS5) | Средняя — нужен при глубоких раскопках, не при ежедневной работе |
| 2 | Связи между фактами | Низкая — для одного проекта можно держать в голове |
| 3 | Автоконсолидация (LLM «сон») | Низкая — агент и так знает что делал |
| 4 | Decay/забывание | Низкая — чекпоинты в архиве не мешают |
| 5 | Версионирование фактов | Низкая — редкий кейс |

### 4.4. Когда схема C достаточна, а когда нужна B

**Схема C достаточна если:**
- Один проект, один активный агент
- Глубина памяти нужна на 5-10 дней
- Поиск по историческим данным — редкий кейс
- Хочется минимум зависимостей

**Схема B нужна если:**
- Несколько проектов / несколько агентов
- Нужен поиск по накопленным знаниям за месяцы
- Факты часто обновляются/противоречат друг другу
- Проект большой (>1000 файлов, сложная архитектура)

### 4.5. Путь эволюции: C → B

```
Схема C                    Схема B (при необходимости)
─────────────────    →    ──────────────────────────
SESSION_SUMMARY.md          MEMORY.md (из SQLite)
чекпоинты (читаются)        чекпоинты (архив)
worklog.md                  JSONL логи
ручное обновление           + query.py (поиск)
                            + sleep.py (автоконсолидация)
```

SESSION_SUMMARY.md при переходе на B становится входными данными для начального заполнения БД.

## 7. Что остаётся без изменений (для всех схем)

| Артефакт | Причина |

## 8. Фазовый план внедрения

### 8.1. Для схемы C (рекомендуемый первый шаг)

| Фаза | Что | Результат | Усилие |
|------|-----|-----------|--------|
| **C0** | Создать `SESSION_SUMMARY.md` вручную | Компактный старт | 10 мин |
| **C1** | Добавить правило в START_PROMPT.md | Агент знает о файле | 2 мин |
| **C2** | Скрипт `gen_summary.py` (без LLM) | Автообновление из чекпоинтов | 2-3 часа |
| **C3** | Скрипт `gen_summary.py --llm` | LLM-суммаризация | +2-3 часа |

**Фаза C0+C1 дают 80% пользы при 5% усилий.**

### 8.2. Для схемы B (при необходимости)

| Фаза | Что | Результат | Усилие |
|------|-----|-----------|--------|
| **B0** | Создать `MEMORY.md` вручную | = фаза C0 | 10 мин |
| **B1** | `memory/query.py` + SQLite + FTS5 | Поиск по памяти | 1-2 дня |
| **B2** | `memory/log.py` + JSONL | Логирование вместо worklog | 1-2 дня |
| **B3** | `memory/sleep.py` — light sleep | Автоимпорт JSONL→SQLite | 1-2 дня |
| **B4** | `memory/sleep.py` — deep sleep | LLM-извлечение фактов | 2-3 дня |
| **B5** | Миграция чекпоинтов | Старые знания в БД | 1 день |

### 8.3. Рекомендуемый путь

```
Начать: C0 + C1 (сегодня, 10 минут)
   ↓
Работать. Оценить, хватает ли SESSION_SUMMARY.md
   ↓
Если не хватает поиска → B1 (query.py + FTS5)
Если не хватает автоконсолидации → B3-B4 (sleep.py)
```

Не обязательно идти до B. Можно остановиться на любом этапе.

## 9. Риски

### 9.1. Схема C

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|------------|---------|-----------|
| R1 | Агент забывает обновить SESSION_SUMMARY.md | Средняя | Среднее | Правило в START_PROMPT.md |
| R2 | Файл устаревает между сессиями | Низкая | Низкое | Обновляется при каждом завершении |
| R3 | Агент пишет слишком много (>500 токенов) | Низкая | Низкое | Лимит в формате |

### 9.2. Схема B

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|------------|---------|-----------|
| R1 | LLM галлюцинирует факты при «сне» | Средняя | Высокое | Confidence < 1.0, is_frozen для критичных |
| R2 | SQLite файл повреждается | Низкая | Высокое | JSONL = source of truth, SQLite пересоздаётся |
| R3 | JSONL конфликты при мерже | Низкая | Низкое | Append-only, авто-мерж |
| R4 | Превышение размера MEMORY.md | Низкая | Среднее | Лимит <500 токенов |
| R5 | Агент не использует query.py | Средняя | Среднее | MEMORY.md покрывает 80%, инструкция в START_PROMPT |
| R6 | Зависимость от LLM для deep sleep | Высокая | Среднее | Light sleep без LLM, deep — опционально |

## 10. Открытые вопросы

1. **Название файла** — `SESSION_SUMMARY.md` или `MEMORY.md`?
2. **Глубина саммари** — 5 дней? 3? 7?
3. **Обновление при схеме C** — только вручную (C1)? Или сразу скрипт (C2)?
4. **Источник эмбеддингов** — для схемы B. Пока отложен, FTS5 достаточно.
5. **Частота deep sleep** — для схемы B. Раз в день? При завершении сессии?
6. **Кто запускает sleep** — сам агент? Внешний cron? Триггер по количеству записей?

## 11. Вывод

**Рекомендация:** Начать со схемы C (чекпоинты + SESSION_SUMMARY.md). Это даёт основную пользу — экономию токенов при старте — с нулевым риском и за 10 минут. Если/когда появятся потребности в поиске и автоконсолидации — плавно перейти к схеме B, используя SESSION_SUMMARY.md как начальные данные для БД.

Схема B — это правильная архитектура для масштабирования. Но сейчас, при одном проекте и одном агенте, схема C — прагматичный максимум пользы при минимуме усилий.
