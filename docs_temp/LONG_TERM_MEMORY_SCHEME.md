# Схема долговременной памяти ИИ-агента: текущая vs предлагаемая

*Создано: 2026-04-22 12:32:47 UTC*

---

## 1. Текущая схема памяти

### 1.1. Артефакты

| Артефакт | Объём | ≈ Токенов | Когда читается | Назначение |
|----------|-------|-----------|----------------|------------|
| `START_PROMPT.md` | 7 KB | ~2 400 | Каждый старт | Правила работы, структура папок, запреты |
| `Caveman.md` | 4 KB | ~1 400 | Каждый старт | Режим коммуникации |
| `worklog.md` | 102 KB | ~34 000 | Каждый старт (хвост) | Хроника работы всех агентов |
| `checkpoints/` (116 файлов) | 642 KB | ~214 000 | Выборочно | Фиксация этапов работы |
| `docs/SORTING_LAYERS.md` | 19 KB | ~6 300 | При необходимости | Документация по слоям |

### 1.2. Как это работает сейчас

```
Старт сессии:
  1. Прочитать START_PROMPT.md         → ~2 400 токенов
  2. Прочитать Caveman.md              → ~1 400 токенов
  3. Прочитать worklog.md (хвост)      → ~3 000-5 000 токенов
  4. Прочитать последние чекпоинты     → ~2 000-5 000 токенов
  ─────────────────────────────────────
  Итого при старте:                    ~8 000-14 000 токенов

В процессе работы:
  - Обновить worklog.md (append)
  - Создать чекпоинт checkpoints/ММ_ДД_цель.md
  - Git push при завершении

Следующая сессия:
  - Контекст из предыдущей сессии ПОТЕРЯН
  - Всё восстанавливается через чтение файлов
  - Чекпоинты неструктурированы — агент не знает, какие релевантны
```

### 1.3. Проблемы текущей схемы

| # | Проблема | Влияние |
|---|---------|---------|
| P1 | **Дорогостоящий старт** — чтение файлов каждый раз | 8-14K токенов тратится до начала работы |
| P2 | **Нет выборочного доступа** — всё или ничего | Агент читает нерелевантные чекпоинты или не читает нужные |
| P3 | **Нет поиска** — только чтение по имени файла | Невозможно найти «решение про орбитальное оружие» без знания имени файла |
| P4 | **Дублирование** — чекпоинты повторяют друг друга | 116 файлов, многие перекрываются по содержимому |
| P5 | **Нет связей** — факты изолированы | Решение в чекпоинте #47 не связано с фактом в #23 |
| P6 | **Нет забывания** — всё копится | 642 KB чекпоинтов, 102 KB worklog, и растёт |
| P7 | **worklog бесполезен после N сессий** | 34K токенов текста, найти что-то — невозможно |
| P8 | **Ручное создание чекпоинтов** | Агент забывает — чекпоинт не создан, контекст потерян |

## 2. Предлагаемая схема памяти

### 2.1. Три уровня (кэш-модель)

```
L1 — MEMORY.md           → Всегда в контексте, <500 токенов
                             Автогенерация при каждом «сне»
                             ЗАМЕНА чтения чекпоинтов при старте

L2 — SQLite + FTS5       → По запросу, когда нужна конкретика
                             python3 memory/query.py "запрос"
                             ~200 токенов на запрос

L3 — JSONL логи          → Source of truth, append-only
                             Один файл на день
                             Git-мержимый (текстовый)
                             SQLite — производное, пересоздаётся
```

### 2.2. Структура файлов

```
/home/z/my-project/
├── MEMORY.md                    # L1 — автогенерация, <500 токенов
├── memory/
│   ├── log/                     # L3 — append-only JSONL (Git-мержимые)
│   │   ├── 2026-04-20.jsonl
│   │   ├── 2026-04-21.jsonl
│   │   └── 2026-04-22.jsonl
│   ├── db/                      # L2 — SQLite (gitignored, производное)
│   │   └── agent_memory.db      # Генерируется из JSONL при старте
│   ├── query.py                 # CLI: поиск по памяти
│   ├── log.py                   # CLI: добавление записи
│   ├── sleep.py                 # CLI: цикл «сна»
│   └── schema.sql               # Схема БД + миграции
├── START_PROMPT.md              # Остаётся (правила работы)
├── Caveman.md                   # Остаётся (режим коммуникации)
└── ...
```

### 2.3. Схема SQLite

```sql
-- Эпизоды (контекст: когда, почему был сделан вывод)
CREATE TABLE episodes (
    id            INTEGER PRIMARY KEY,
    session_id    TEXT NOT NULL,        -- ID сессии агента
    date          TEXT NOT NULL,        -- YYYY-MM-DD
    trigger_type  TEXT NOT NULL,        -- user_request | auto | sleep
    summary       TEXT,                 -- Краткое описание сессии
    created_at    TEXT NOT NULL
);

-- Факты (атомарные утверждения о проекте)
CREATE TABLE facts (
    id              INTEGER PRIMARY KEY,
    content         TEXT NOT NULL,
    type            TEXT NOT NULL,      -- knowledge | constraint | dependency | warning
    confidence      REAL DEFAULT 0.8,  -- 0.0-1.0
    valid_from      TEXT NOT NULL,
    valid_until     TEXT,               -- NULL = актуален
    superseded_by   INTEGER REFERENCES facts(id),
    source_episode  INTEGER REFERENCES episodes(id),
    access_count    INTEGER DEFAULT 0,
    last_accessed   TEXT,
    is_active       INTEGER DEFAULT 1,
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL
);

-- Решения (архитектурные/технические выборы)
CREATE TABLE decisions (
    id            INTEGER PRIMARY KEY,
    what          TEXT NOT NULL,        -- Что решено
    why           TEXT NOT NULL,        -- Почему так
    alternatives  TEXT,                 -- Какие варианты рассматривались
    is_frozen     INTEGER DEFAULT 0,   -- Заморожено (не менять без подтверждения)
    source_episode INTEGER REFERENCES episodes(id),
    created_at    TEXT NOT NULL
);

-- Задачи
CREATE TABLE tasks (
    id            INTEGER PRIMARY KEY,
    title         TEXT NOT NULL,
    description   TEXT,
    status        TEXT NOT NULL,        -- active | completed | blocked | archived
    parent_task   INTEGER REFERENCES tasks(id),
    source_episode INTEGER REFERENCES episodes(id),
    created_at    TEXT NOT NULL,
    updated_at    TEXT NOT NULL
);

-- Связи между сущностями (граф знаний)
CREATE TABLE entity_links (
    source_type   TEXT NOT NULL,        -- fact | decision | task
    source_id     INTEGER NOT NULL,
    target_type   TEXT NOT NULL,
    target_id     INTEGER NOT NULL,
    relation      TEXT NOT NULL,        -- depends_on | justifies | contradicts |
                                        -- supersedes | blocks | relates_to
    created_at    TEXT NOT NULL,
    PRIMARY KEY (source_type, source_id, target_type, target_id, relation)
);

-- FTS5 полнотекстовый поиск
CREATE VIRTUAL TABLE facts_fts USING fts5(content, type, content=facts, content_rowid=id);
CREATE VIRTUAL TABLE decisions_fts USING fts5(what, why, content=decisions, content_rowid=id);
CREATE VIRTUAL TABLE tasks_fts USING fts5(title, description, content=tasks, content_rowid=id);

-- Триггеры синхронизации FTS5
CREATE TRIGGER facts_fts_insert AFTER INSERT ON facts BEGIN
    INSERT INTO facts_fts(rowid, content, type) VALUES (new.id, new.content, new.type);
END;
CREATE TRIGGER facts_fts_delete AFTER DELETE ON facts BEGIN
    INSERT INTO facts_fts(facts_fts, rowid, content, type) VALUES('delete', old.id, old.content, old.type);
END;
CREATE TRIGGER facts_fts_update AFTER UPDATE ON facts BEGIN
    INSERT INTO facts_fts(facts_fts, rowid, content, type) VALUES('delete', old.id, old.content, old.type);
    INSERT INTO facts_fts(rowid, content, type) VALUES (new.id, new.content, new.type);
END;
-- Аналогичные триггеры для decisions_fts и tasks_fts

-- Метаданные
CREATE TABLE metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Записи: schema_version, last_sleep_at, last_rebuild_at
```

### 2.4. Формат JSONL логов

Каждая строка — JSON-объект, одна запись:

```jsonl
{"ts":"2026-04-22T12:30:00Z","type":"fact","data":{"content":"ProjectSettings вызывает краш Unity Editor","type":"constraint","confidence":1.0}}
{"ts":"2026-04-22T12:31:00Z","type":"decision","data":{"what":"Удалить ProjectSettings из репо","why":"Краш Unity Editor при загрузке","alternatives":"Попытаться исправить формат файлов","is_frozen":true}}
{"ts":"2026-04-22T12:32:00Z","type":"task","data":{"title":"Проверка SORTING_LAYERS.md","status":"active"}}
{"ts":"2026-04-22T12:33:00Z","type":"task_update","data":{"task_id":3,"status":"completed"}}
{"ts":"2026-04-22T12:34:00Z","type":"link","data":{"source_type":"decision","source_id":2,"target_type":"fact","target_id":1,"relation":"justifies"}}
```

### 2.5. MEMORY.md — автогенерация (L1)

Формат, пересоздаваемый при каждом deep sleep:

```markdown
# Активный контекст (автогенерация, НЕ редактировать вручную)
Обновлено: 2026-04-22 12:30 UTC

## Проект
```

**~180 токенов.** Вместо 8-14K при текущей схеме.

### 2.6. Цикл «сна»

```
┌───────────────┐     ┌─────────────────┐     ┌──────────────────┐
│  Light sleep   │────▶│   Deep sleep    │────▶│    Snapshot      │
│  (без LLM)    │     │    (с LLM)      │     │   MEMORY.md      │
│                │     │                 │     │                  │
│  • JSONL→SQLite│     │  • LLM-извлечение│    │  • Автогенерация │
│  • FTS5 индекс │     │    фактов       │     │    из БД         │
│  • Дедупликация│     │  • Связывание   │     │  • <500 токенов  │
│                │     │  • Decay        │     │                  │
│  Триггер:      │     │  Триггер:       │     │  При каждом      │
│  каждые N      │     │  раз в день /   │     │  deep sleep      │
│  действий      │     │  конец сессии   │     │                  │
└───────────────┘     └─────────────────┘     └──────────────────┘
```

**Light sleep** (без LLM, мгновенный):
- Импорт новых JSONL записей в SQLite
- Обновление FTS5-индексов
- Пометка обработанных логов

**Deep sleep** (с LLM, раз в день / при завершении сессии):
- LLM анализирует сырые логи диалогов и действий
- Извлекает факты, решения, обновляет задачи
- Дедупликация по FTS5 (близкие тексты → merge)
- Decay: факты с confidence < 0.3 и last_accessed > 30 дней → is_active = false
- Генерация MEMORY.md из текущего состояния БД
- Git add + commit + push (JSONL + MEMORY.md)

### 2.7. Рабочий процесс агента

```
Старт сессии:
  1. Прочитать MEMORY.md (~200 токенов)     ← ЗАМЕНА чекпоинтов
  2. Прочитать START_PROMPT.md (~2 400)     ← обязательно, правила
  3. Прочитать Caveman.md (~1 400)          ← обязательно, режим
  ─────────────────────────────────────
  Итого при старте: ~4 000 токенов (вместо 8-14K)

Во время работы:
  Нужна конкретика?  → python3 memory/query.py "инвентарь слоты"
                       → 3-5 фактов ~200 токенов
  Принял решение?    → python3 memory/log.py decision "текст" "почему"
  Нашёл факт?        → python3 memory/log.py fact "текст"
  Обновил задачу?    → python3 memory/log.py task_update ID статус

Конец сессии:
  → python3 memory/sleep.py --deep
  → LLM извлекает факты, обновляет MEMORY.md
  → git add + commit + push
```

## 3. Сравнение: что теряем, что получаем

### 3.1. Что ПОЛУЧАЕМ

| # | Преимущество | Описание | Оценка |
|---|-------------|----------|--------|
| G1 | **Экономия токенов при старте** | 4K вместо 8-14K (–50-70%) | ⭐⭐⭐ |
| G2 | **Выборочный поиск** | FTS5: найти релевантное без чтения всего | ⭐⭐⭐ |
| G3 | **Структурированные знания** | Факты, решения, задачи — вместо свободного текста | ⭐⭐⭐ |
| G4 | **Связи между сущностями** | Граф: решение→факт→задача | ⭐⭐ |
| G5 | **Автоматическая консолидация** | LLM извлекает факты из логов при «сне» | ⭐⭐⭐ |
| G6 | **Забывание устаревшего** | Decay + is_active, БД не растёт бесконечно | ⭐⭐ |
| G7 | **Человекочитаемый срез** | MEMORY.md — всегда актуальное саммари | ⭐⭐ |
| G8 | **Git-мержимость** | JSONL — текст, мержится без конфликтов | ⭐⭐ |
| G9 | **Версионирование фактов** | superseded_by — эволюция знаний во времени | ⭐ |
| G10 | **Эпизодический контекст** | Почему факт считается верным (source_episode) | ⭐⭐ |

### 3.2. Что ТЕРЯЕМ

| # | Потеря | Описание | Серьёзность | Компенсация |
|---|-------|----------|-------------|-------------|
| L1 | **Простота** | Текущая схема = файлы, ничего не надо ставить | Средняя | SQLite есть в Python стандартно, FTS5 тоже |
| L2 | **Полный текст чекпоинтов** | Чекпоинты содержат контекст + ход мыслей | Низкая | LLM-извлечение при «сне» сохранит суть. Сырые чекпоинты остаются в Git истории |
| L3 | **Нулевая зависимость** | Появляются Python-скрипты, SQLite | Низкая | Python уже есть в окружении, SQLite стандартный |
| L4 | **Немедленная доступность** | Чекпоинт = файл, открыл и читаешь. SQLite = нужен запрос | Средняя | MEMORY.md даёт 80% пользы без запросов. query.py для оставшихся 20% |
| L5 | **Гарантия консистентности** | Чекпоинт написан — он точен. LLM при «сне» может галлюцинировать | Средняя | Confidence для фактов. Человек может просмотреть MEMORY.md. Флаг is_frozen для критичных решений |
| L6 | **Привычный формат** | Вся команда привыкла к чекпоинтам | Низкая | MEMORY.md читается так же. JSONL — для машины, не для людей |

### 3.3. Риски

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|------------|---------|-----------|
| R1 | LLM галлюцинирует факты при «сне» | Средняя | Высокое | Confidence < 1.0 для извлечённых фактов. Флаги is_frozen для проверенных человеком. Векторная дедупликация снизит дубли |
| R2 | SQLite файл повреждается | Низкая | Высокое | JSONL = source of truth, SQLite пересоздаётся. Vacuum/pragma integrity_check при rebuild |
| R3 | JSONL конфликты при мерже | Низкая | Низкое | Append-only формат, разные строки — автоматический мерж. Конфликт только при пересечении дат |
| R4 | Превышение размера MEMORY.md | Низкая | Среднее | Жёсткий лимим <500 токенов. Активные задачи — топ-10. Предупреждения — только незамороженные |
| R5 | Отказ агента использовать query.py | Средняя | Среднее | MEMORY.md покрывает 80% кейсов. Явная инструкция в START_PROMPT.md: «для деталей используй memory/query.py» |
| R6 | Зависимость от LLM для deep sleep | Высокая | Среднее | Light sleep работает без LLM. Deep sleep — опционально, можно запускать вручную |
