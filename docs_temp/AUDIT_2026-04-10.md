## 🔴 Критические проблемы (P0)

### C-01: `Rigidbody2D.velocity` устарел в Unity 6
**Файл:** `PlayerController.cs:223`  
**Код:** `rb.velocity = moveInput * speed;`  
**Проблема:** В Unity 6 `Rigidbody2D.velocity` заменён на `Rigidbody2D.linearVelocity`. Текущий код вызовет предупреждение об устаревшем API и может вести себя непредсказуемо.  
**Исправление:** `rb.linearVelocity = moveInput * speed;`

### C-02: `DefenseProcessor` использует `System.Random` вместо `UnityEngine.Random`
**Файл:** `DefenseProcessor.cs:77,232,271`  
**Код:** `private static readonly Random random = new Random();`  
**Проблема:** `System.Random` не связан с Unity seed системой. При воспроизведении боев (replay/тесты) результаты будут невоспроизводимы. Unity-тесты могут получать разные результаты.  
**Исправление:** Заменить на `UnityEngine.Random` или передавать `SeededRandom` из проекта.

### C-03: Сердце не должно иметь чёрную HP ← Chat:C1
**Файл:** `BodyPart.cs:93-95`  
**Код:** 
```csharp
this.maxBlackHP = maxRedHP * GameConstants.STRUCTURAL_HP_MULTIPLIER;
this.currentBlackHP = this.maxBlackHP;
```
**Проблема:** Комментарий на строке 92 явно указывает: "Для сердца должна быть только красная HP (blackHP = 0)". Но реализация устанавливает blackHP для ВСЕХ частей, включая сердце. Это нарушает BODY_SYSTEM.md.  
**Исправление:** Добавить условие: если `partType == BodyPartType.Heart`, установить `maxBlackHP = 0`.  
**Статус верификации:** ✅ Подтверждено в коде

### C-04: `ICombatant.SpendQi` принимает `int`, но QiController использует `long`
**Файл:** `Combatant.cs:62`, `QiController.cs:162`  
**Проблема:** `ICombatant.SpendQi(int amount)` при уровне культивации L5+ будет переполнен (Qi > 2.1 млрд). `QiController.CurrentQi` имеет тип `long`, но интерфейс ограничен `int`. Это приведёт к невозможности потратить Ци на техники для высокоуровневых практиков.  
**Исправление:** Изменить `SpendQi(int)` на `SpendQi(long)` в интерфейсе и всех реализациях.

### C-05: `CombatantBase.GetDefenderParams` сужает `CurrentQi` до `int` ← Chat:C5
**Файл:** `Combatant.cs:275`  
**Код:** `CurrentQi = (int)Math.Min(CurrentQi, int.MaxValue)`  
**Проблема:** При высоких уровнях культивации Qi > int.MaxValue, DefenderParams.CurrentQi (int) будет переполнен. QiBuffer получит некорректные данные. Для L7+ культиваторов Ци превышает int.MaxValue (~2.1 млрд).  
**Исправление:** Изменить `DefenderParams.CurrentQi` на `long`.  
**Статус верификации:** ✅ Подтверждено в коде

### C-06: `PlayerController` использует старый Input API
**Файл:** `PlayerController.cs:23,174-197`  
**Код:** `using UnityEngine.InputSystem;` + `Keyboard.current`  
**Проблема:** Новый Input System используется, но НЕ проверяется `Keyboard.current != null` перед доступом. Однако если Input System не установлена в проекте, код не скомпилируется.  
**Статус:** Проверить manifest.json — если `com.unity.inputsystem` отсутствует, код не скомпилируется.

### C-07: `StatDevelopment` — не MonoBehaviour, но имеет `[SerializeField]` ← Chat:H2
**Файл:** `StatDevelopment.cs:86-88,126-135`  
**Проблема:** Класс `StatDevelopment` — обычный C# класс (не наследует MonoBehaviour), но использует `[SerializeField]`. В Unity `[SerializeField]` работает ТОЛЬКО для MonoBehaviour и ScriptableObject. Поля НЕ будут сериализованы Inspector.  
**Исправление:** Либо сделать StatDevelopment MonoBehaviour, либо убрать `[SerializeField]` и использовать `[Serializable]` поле в родительском MonoBehaviour.  
**Статус верификации:** ✅ Подтверждено в коде

### C-08: `StatDevelopment` — характеристики ниже 10 не могут расти ← Chat:C2
**Файл:** `StatDevelopment.cs:322-333`  
**Код:**
```csharp
public float GetThreshold(StatType type)
{
    float currentStat = GetStat(type);
    return Mathf.Floor(currentStat / 10f);
}
public bool CanAdvance(StatType type)
{
    float delta = GetDelta(type);
    float threshold = GetThreshold(type);
    return delta >= threshold && threshold > 0f;
}
```
**Проблема:** Формула `threshold = floor(currentStat / 10)`. При stat=9, threshold=0. `CanAdvance()` возвращает false когда `threshold <= 0f`. Это блокирует развитие ВСЕХ характеристик ниже 10. Ни одна характеристика ниже 10 никогда не сможет вырасти.  
**Исправление:** `threshold = Math.Max(1f, Mathf.Floor(currentStat / 10f))` — гарантировать минимальный порог = 1.  
**Статус верификации:** ✅ Подтверждено в коде

### C-09: `ParryBonus`/`BlockBonus` двойной расчёт ← Chat:C4
**Файл:** `Combatant.cs:282-284`, `DamageCalculator.cs:192-194`  
**Код:**
```csharp
// Combatant.cs GetDefenderParams():
ParryBonus = ParryChance,   // ParryChance = уже финальный шанс
BlockBonus = BlockChance,   // BlockChance = уже финальный шанс

// DamageCalculator.cs:
ParryChance = DefenseProcessor.CalculateParryChance(defender.Agility, defender.ParryBonus),
```
**Проблема:** `ParryChance` property уже вызывает `DefenseProcessor.CalculateParryChance(agility, bonus)` и возвращает финальный шанс (базовый + бонус от AGI). Но `GetDefenderParams()` передаёт этот финальный шанс как `ParryBonus` в `DefenderParams`. Затем `DamageCalculator` снова вызывает `CalculateParryChance(agility, parryBonus)` — парирование рассчитывается ДВАЖДЫ: шанс = базовый_расчёт + финальный_шанс. Аналогично для BlockChance/BlockBonus.  
**Исправление:** Передавать сырые бонусы экипировки (weaponParryBonus, shieldBlockBonus), а не рассчитанные финальные шансы.  
**Статус верификации:** ✅ Подтверждено в коде

### C-10: `CombatLog` подписывается на статическое событие в статическом конструкторе ← Chat:H4
**Файл:** `CombatEvents.cs:274-277`  
**Код:** 
```csharp
static CombatLog()
{
    CombatEvents.OnCombatEvent += AddEntry;
}
```
**Проблема:** Статический конструктор может не выполниться, если к CombatLog никогда не обращаются напрямую. В этом случае лог боя не будет записываться. Также нет отписки — утечка памяти.  
**Исправление:** Инициализировать подписку явно (при старте игры) или через ленивую инициализацию. Добавить метод очистки и вызывать при смене сцены.  
**Статус верификации:** ✅ Подтверждено в коде

### C-11: `DamageCalculator` игнорирует стихийные взаимодействия ← Chat:C3 (повышено с H-01 до CRITICAL)
**Файл:** `DamageCalculator.cs:307-314`  
**Код:**
```csharp
private static float CalculateElementMultiplier(Element element)
{
    if (element == Element.Void) return GameConstants.VOID_ELEMENT_MULTIPLIER;
    if (element == Element.Neutral) return 1.0f;
    return 1.0f;  // ВСЕ элементы = 1.0!
}
```
**Проблема:** Метод `CalculateElementMultiplier` возвращает 1.0 для ВСЕХ элементов кроме Void. Метод `CalculateElementalInteraction` (строка 325) реализован корректно с противоположными элементами (Fire↔Water, Earth↔Air ×1.5), но НИГДЕ НЕ ВЫЗЫВАЕТСЯ. Это полностью ломает стихийную систему боя — одна из ключевых механик игры.  
**Исправление:** Заменить `CalculateElementMultiplier(element)` на вызов `CalculateElementalInteraction(attackerElement, defenderElement)` и добавить параметр defender element в пайплайн расчёта урона.  
**Статус верификации:** ✅ Подтверждено в коде — CalculateElementalInteraction существует, но не вызывается
