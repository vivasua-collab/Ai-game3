## 🔴 Критические проблемы (P0)

### C-01: `Rigidbody2D.velocity` устарел в Unity 6
**Файл:** `PlayerController.cs:223`  
**Код:** `rb.velocity = moveInput * speed;`  
**Проблема:** В Unity 6 `Rigidbody2D.velocity` заменён на `Rigidbody2D.linearVelocity`. Текущий код вызовет предупреждение об устаревшем API и может вести себя непредсказуемо.  
**Исправление:** `rb.linearVelocity = moveInput * speed;`

### C-02: `DefenseProcessor` использует `System.Random` вместо `UnityEngine.Random`
**Файл:** `DefenseProcessor.cs:77,232,271`  
**Код:** `private static readonly Random random = new Random();`  
**Проблема:** `System.Random` не связан с Unity seed системой. При воспроизведении боев (replay/тесты) результаты будут невоспроизводимы. Unity-тесты могут получать разные результаты.  
**Исправление:** Заменить на `UnityEngine.Random` или передавать `SeededRandom` из проекта.

### C-03: Сердце не должно иметь чёрную HP
**Файл:** `BodyPart.cs:93-95`  
**Код:** 
```csharp
this.maxBlackHP = maxRedHP * GameConstants.STRUCTURAL_HP_MULTIPLIER;
this.currentBlackHP = this.maxBlackHP;
```
**Проблема:** Комментарий на строке 92 явно указывает: "Для сердца должна быть только красная HP (blackHP = 0)". Но реализация устанавливает blackHP для ВСЕХ частей, включая сердце. Это нарушает BODY_SYSTEM.md.  
**Исправление:** Добавить условие: если `partType == BodyPartType.Heart`, установить `maxBlackHP = 0`.

### C-04: `ICombatant.SpendQi` принимает `int`, но QiController использует `long`
**Файл:** `Combatant.cs:62`, `QiController.cs:162`  
**Проблема:** `ICombatant.SpendQi(int amount)` при уровне культивации L5+ будет переполнен (Qi > 2.1 млрд). `QiController.CurrentQi` имеет тип `long`, но интерфейс ограничен `int`. Это приведёт к невозможности потратить Ци на техники для высокоуровневых практиков.  
**Исправление:** Изменить `SpendQi(int)` на `SpendQi(long)` в интерфейсе и всех реализациях.

### C-05: `CombatantBase.GetDefenderParams` сужает `CurrentQi` до `int`
**Файл:** `Combatant.cs:275`  
**Код:** `CurrentQi = (int)Math.Min(CurrentQi, int.MaxValue)`  
**Проблема:** При высоких уровнях культивации Qi > int.MaxValue, DefenderParams.CurrentQi (int) будет переполнен. QiBuffer получит некорректные данные.  
**Исправление:** Изменить `DefenderParams.CurrentQi` на `long`.

### C-06: `PlayerController` использует старый Input API
**Файл:** `PlayerController.cs:23,174-197`  
**Код:** `using UnityEngine.InputSystem;` + `Keyboard.current`  
**Проблема:** Новый Input System используется, но НЕ проверяется `Keyboard.current != null` перед доступом (строка 174 есть проверка, но на 193 строке `Keyboard.current.f5Key.wasPressedThisFrame` внутри того же блока — ОК). Однако если Input System не установлена в проекте, код не скомпилируется. Пакет должен быть добавлен в manifest.json.  
**Статус:** Проверить manifest.json — если `com.unity.inputsystem` отсутствует, код не скомпилируется.

### C-07: `StatDevelopment` — не MonoBehaviour, но имеет `[SerializeField]`
**Файл:** `StatDevelopment.cs:86-88,126-135`  
**Проблема:** Класс `StatDevelopment` — обычный C# класс (не наследует MonoBehaviour), но использует `[SerializeField]`. В Unity `[SerializeField]` работает ТОЛЬКО для MonoBehaviour и ScriptableObject. Поля НЕ будут сериализованы Inspector.  
**Исправление:** Либо сделать StatDevelopment MonoBehaviour, либо убрать `[SerializeField]` и использовать `[Serializable]` поле в родительском MonoBehaviour.

### C-08: `CombatLog` подписывается на статическое событие в статическом конструкторе
**Файл:** `CombatEvents.cs:274-277`  
**Код:** 
```csharp
static CombatLog()
{
    CombatEvents.OnCombatEvent += AddEntry;
}
```
**Проблема:** Статический конструктор может не выполниться, если к CombatLog никогда не обращаются напрямую. В этом случае лог боя не будет записываться. Также нет отписки — утечка памяти.  
**Исправление:** Инициализировать подписку явно (при старте игры) или через ленивую инициализацию.

## 🟠 Высокие проблемы (P1)

### H-01: `DamageCalculator.CalculateElementMultiplier` не учитывает элемент защитника
**Файл:** `DamageCalculator.cs:307-314`  
**Проблема:** Метод принимает только `Element attacker`, но не defender. Противоположные элементы не рассчитываются. Хотя `CalculateElementalInteraction` реализован корректно, `CalculateElementMultiplier` (используемый в пайплайне) всегда возвращает 1.0 или VOID множитель.  
**Исправление:** Добавить параметр `Element defenderElement` и использовать `CalculateElementalInteraction`.

### H-02: Duplicate `PlayerSaveData` class
**Файлы:** `Save/SaveManager.cs:532` и `Player/PlayerController.cs:496`  
**Проблема:** Два разных класса `PlayerSaveData` с разными полями. SaveManager.PlayerSaveData имеет 4 поля, PlayerController.PlayerSaveData — 6 полей. При сериализации/десериализации возможна потеря данных.  
**Исправление:** Объединить в один класс в `SaveDataTypes.cs`.

### H-03: `SaveManager` использует `FindFirstObjectByType` вместо ServiceLocator
**Файл:** `SaveManager.cs:104-108`  
**Проблема:** Все 5 вызовов `FindFirstObjectByType` — дорогие операции O(n). При первом запуске Acceptable, но при повторных — ServiceLocator быстрее O(1).  
**Исправление:** Использовать `ServiceLocator.GetOrFind<T>()`.

### H-04: `VFXPool` — максимальный размер пула глобальный, а не на префаб
**Файл:** `VFXPool.cs:37,113`  
**Проблема:** `maxPoolSize = 50` ограничивает ВСЕ объекты в пуле суммарно, а не каждый тип префаба отдельно. Если есть 10 типов VFX, каждый получит ~5 объектов — этого мало.  
**Исправление:** Изменить на per-prefab пул с индивидуальным лимитом.

### H-05: `GameInitializer.SubscribeToEvents()` никогда не вызывается
**Файл:** `GameInitializer.cs:356-364`  
**Проблема:** Метод `SubscribeToEvents()` существует, но в `InitializeGameAsync()` вызываются только `SubscribeToSaveEvents()` и `SubscribeToPlayerEvents()` отдельно. `SubscribeToTimeEvents()` НЕ вызывается нигде в инициализации.  
**Исправление:** Заменить раздельные вызовы на `SubscribeToEvents()`.

### H-06: `GameEvents` — статические события без отписки при смене сцены
**Файл:** `GameEvents.cs` (весь)  
**Проблема:** Все события — статические. При загрузке новой сцены MonoBehaviour объекты отписываются, но если подписка была через lambda — утекает. `ClearAllEvents()` существует, но не вызывается автоматически.  
**Исправление:** Вызывать `GameEvents.ClearAllEvents()` при смене сцены.

### H-07: `RegenerationMultipliers[9]` = `float.PositiveInfinity`
**Файл:** `Constants.cs:217`  
**Проблема:** L10 регенерация = PositiveInfinity. Умножение на Time.deltaTime даёт Infinity. `AddQi((long)Infinity)` приведёт к неопределённому поведению.  
**Исправление:** Заменить на `float.MaxValue` и обрабатывать как "мгновенное полное восстановление".

### H-08: `BodyPartHitChances` — Dictionary с enum ключом не гарантирует порядок
**Файл:** `DefenseProcessor.cs:230-245`, `Constants.cs:428-441`  
**Проблема:** `RollBodyPart()` итерирует `GameConstants.BodyPartHitChances` через foreach. Dictionary не гарантирует порядок. Если порядок изменится, cumulative sum будет некорректной.  
**Исправление:** Использовать массив с фиксированным порядком или `OrderBy`.

### H-09: `QiController.Meditate()` — не передаёт время в игровые часы
**Файл:** `QiController.cs:248-260`  
**Проблема:** Медитация принимает `durationTicks`, но не продвигает игровое время через `TimeController`. Игрок получает Ци "бесплатно" без траты игрового времени.  
**Исправление:** Добавить продвижение времени: `timeController.AdvanceHours(ticks / 60)`.

### H-10: Нет `ICombatant` реализации на PlayerController
**Файл:** `PlayerController.cs`  
**Проблема:** `PlayerController` не наследует `CombatantBase` и не реализует `ICombatant`. CombatManager не может работать с игроком напрямую.  
**Исправление:** Либо PlayerController : CombatantBase, либо добавить ICombatant.

### H-11: `GameManager.FindReferences()` в Awake — гонка инициализации
**Файл:** `GameManager.cs:155-165`  
**Проблема:** `FindFirstObjectByType` в Awake может не найти объекты, которые ещё не инициализированы (порядок Awake не гарантирован).  
**Исправление:** Использовать отложенную инициализацию или ServiceLocator.Request<T>().

### H-12: `CombatManager` не отписывается от `OnDeath` при завершении боя корректно
**Файл:** `CombatManager.cs:379-383`  
**Проблема:** В `EndCombat()` отписка происходит через `foreach (var combatant in combatants)`, но если combatant был уничтожен (Destroy) между началом и концом боя, `combatant.OnDeath -= HandleCombatantDeath` может упасть.  
**Исправление:** Добавить null check перед отпиской.

## Кросс-системные проблемы

1. **int vs long для Qi** — ICombatant, DefenderParams, QiBuffer работают с int, но QiController с long. Система треснёт на L5+ уровнях. (C-04, C-05)

2. **ICombatant не реализован на Player** — CombatManager может работать с NPC, но не с игроком. (H-10)

3. **Save система неполная** — Сохраняются только Time, World, Location, Faction, Event. Player, Qi, Body, NPC, Combat — НЕ сохраняются.

4. **Дублирование событий** — GameEvents (статические), CombatEvents (статические), локальные события на MonoBehaviour. Три параллельные системы событий без координации.

5. **Input System** — Смешивание старого (GameManager) и нового (PlayerController). Нет единого Input Controller.
