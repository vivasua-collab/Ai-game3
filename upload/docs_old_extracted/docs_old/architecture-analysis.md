> **Версия:** 1.0 | **Принцип:** Память первична, БД вторична

### Иерархия источников истины

```
┌─────────────────────────────────────────────────────────────────┐
│                   ИЕРАРХИЯ ИСТОЧНИКОВ ИСТИНЫ                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. ПАМЯТЬ (TruthSystem) ────────────────── ПЕРВИЧНЫЙ         │
│      │                                                          │
│      ├─ Активная сессия                                         │
│      ├─ Все расчёты происходят здесь                            │
│      └─ Мгновенный доступ к данным                              │
│                                                                 │
│   2. БД (Prisma/SQLite) ─────────────────── ВТОРИЧНЫЙ          │
│      │                                                          │
│      ├─ Загрузка сессии при старте                              │
│      ├─ Сохранение при смене локации                            │
│      ├─ Немедленное сохранение критических данных               │
│      └─ Периодическое автосохранение                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Критические события (немедленное сохранение)

| Событие | Действие |
|---------|----------|
| Получение новой техники | `db.characterTechnique.create()` + memory |
| Получение нового предмета | `db.inventoryItem.create()` + memory |
| Смена локации | Сохранение текущего состояния → загрузка новой |
| Прорыв уровня | `applyBreakthrough()` - полное сохранение персонажа |
| Медитация на проводимость | `updateConductivity()` - сохранение в БД |
| Завершение боя | Сохранение состояния персонажа |

## 🚌 Event Bus (Шина событий)

> **Версия:** 3.0 | **Назначение:** Phaser ↔ Server

### Принцип работы

Event Bus используется **ТОЛЬКО** для связи между:
- **Phaser Engine (клиент)** ↔ **Server (Truth System)**

**React компоненты взаимодействуют между собой напрямую через API вызовы!**

### Архитектура

```
┌─────────────────────────────────────────────────────────────────┐
│                        PHASER ENGINE                             │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 1. Проверка Qi локально (быстрая проверка UI)             │  │
│  │ 2. Отправка technique:use через Event Bus                 │  │
│  │ 3. Получение ответа (canUse, damageMultiplier, currentQi) │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────┬───────────────────────────────────┘
                              │ HTTP POST /api/game/event
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        SERVER (Event Bus)                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ handlers/combat.ts → handleTechniqueUse()                 │  │
│  │   ├─ Поиск техники в БД (technique.qiCost)                │  │
│  │   ├─ Проверка владения техникой                           │  │
│  │   ├─ Проверка Qi через TruthSystem                        │  │
│  │   ├─ Списание Qi: TruthSystem.spendQi()                   │  │
│  │   ├─ Расчёт бонусов урона (mastery, conductivity)         │  │
│  │   └─ Возврат: canUse, damageMultiplier, currentQi         │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### Типы событий

| Категория | События |
|-----------|---------|
| **Бой** | `technique:use`, `body:damage`, `body:heal` |
| **Тело** | `body:attach_limb`, `body:regenerate`, `body:update` |
| **Инвентарь** | `inventory:equip_item`, `inventory:unequip_item`, `inventory:drop_item` |

### Важные правила

| Правило | Причина |
|---------|---------|
| **Qi списывается ОДИН раз** | Только в `TruthSystem.spendQi()` |
| **qiCost из БД** | Не хардкодить значения! |
| **changes.currentQi** | Только для информации клиенту |
| **Event Bus ≠ React** | React компоненты → прямые API |

## ⏱️ Система тиков времени

> **Принцип:** 1 тик = 1 минута игрового времени

### Время течёт

```
┌─────────────────────────────────────────────────────────────────────┐
│                     ДЕЙСТВИЯ С ТОЧКОЙ ВРЕМЕНИ                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Движение          move_tile: 1 тик (1 минута)                     │
│   Атака             basic_attack: 1 тик                             │
│   Ход боя           combat_turn: 2 тика                             │
│   Медитация         30-480 тиков (30 мин - 8 часов)                 │
│   Прорыв            breakthrough: 60 тиков (1 час)                  │
│   Разговор          talk: 5 тиков                                   │
│   Сбор ресурсов     gather: 10 тиков                                │
│                                                                     │
│   [ПЛАНИРУЕТСЯ] Пробел — промотка времени                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Время на паузе

```
┌─────────────────────────────────────────────────────────────────────┐
│                     ДЕЙСТВИЯ БЕЗ ТОЧКИ ВРЕМЕНИ                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ⏸️ view_inventory   — Просмотр инвентаря                          │
│   ⏸️ view_techniques  — Просмотр техник                             │
│   ⏸️ view_skills      — Просмотр навыков                            │
│   ⏸️ view_status      — Просмотр статуса                            │
│   ⏸️ view_map         — Просмотр карты                              │
│   ⏸️ open_menu        — Открытие меню                               │
│   ⏸️ pause            — Пауза                                       │
│   ⏸️ БЕЗДЕЙСТВИЕ      — Игрок ничего не делает                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Влияние на архитектуру

| Аспект | Традиционный подход | С тиками |
|--------|---------------------|----------|
| **FPS** | 60 FPS = время течёт | 60 FPS ≠ время течёт |
| **Движение** | Плавное, время течёт | Плавное, тики списываются |
| **Коллизии** | Каждый кадр | При смене позиции |
| **Бездействие** | Время течёт | Время НА ПАУЗЕ |
| **UI** | Время течёт | Время НА ПАУЗЕ |

## 🔒 Sandbox ограничения

> **Критическое ограничение:** Приложение работает в sandboxed iframe

```
┌─────────────────────────────────────────────────────────────────┐
│                    SANDBOX RESTRICTIONS                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ❌ localStorage     → SecurityError: Forbidden               │
│   ❌ sessionStorage   → SecurityError: Forbidden               │
│   ❌ IndexedDB        → SecurityError (вероятно)               │
│   ❌ document.cookie  → SecurityError                           │
│                                                                 │
│   ✅ fetch()          → Работает                                │
│   ✅ WebSocket        → Работает                                │
│   ✅ Web Workers      → Работает                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Решение: Server-only Storage

| Компонент | Решение |
|-----------|---------|
| Session ID | `/api/game/last-session` |
| Saves List | `/api/game/save` (GET) |
| Settings | Server-side (Prisma) |
| Game State | TruthSystem (memory) + Prisma (persist) |

## 🏥 Система тела (Kenshi-style)

> **Версия:** 1.0 | **Файл:** `src/lib/game/body-system.ts`

### Структура части тела

```typescript
interface BodyPart {
  id: string;                    // Уникальный ID
  type: BodyPartType;            // head, torso, arm_left, leg_right, etc.
  functionalHP: number;          // Функциональное здоровье (0-100)
  structuralHP: number;          // Структурная целостность (0-100)
  status: LimbStatus;            // healthy | damaged | crippled | severed
}
```

### Поток повреждения

```
┌──────────────────────────────────────────────────────────────────┐
│                     ПОВРЕЖДЕНИЕ ЧАСТИ ТЕЛА                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   1. ДВИЖОК (Phaser)                                             │
│      └─ Event: body:damage { partId, damage, damageType }        │
│                                                                  │
│   2. EVENT BUS HANDLER (handlers/body.ts)                        │
│      ├─ Загрузка bodyState из БД                                 │
│      ├─ applyDamageToLimb(part, damage, { penetration })         │
│      │   ├─ Функциональная HP уменьшается первой                 │
│      │   ├─ Структурная HP поглощает остаток                     │
│      │   └─ Проверка отрубания (structuralHP <= 0)               │
│      ├─ Обновление статуса части                                 │
│      ├─ Сохранение bodyState в БД                                │
│      └─ Response: { severed, fatal, newStatus }                  │
│                                                                  │
│   3. ДВИЖОК (получение ответа)                                   │
│      ├─ visual:update_body_part → обновление HP баров            │
│      ├─ visual:show_notification → "Отрублено: Левая рука!"      │
│      └─ game:death → если fatal = true                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 📊 Phaser 3 интеграция

### Конфигурация

```typescript
const config: Phaser.Types.Core.GameConfig = {
  type: Phaser.AUTO,
  width: 900,
  height: 550,
  physics: {
    default: 'arcade',
    arcade: { gravity: { x: 0, y: 0 } }
  },
  scene: [LocationScene, TrainingScene, WorldScene],
  scale: {
    mode: Phaser.Scale.FIT,
    autoCenter: Phaser.Scale.CENTER_BOTH
  }
};
```

### Сцены

| Сцена | Назначение |
|-------|------------|
| `LocationScene` | Основная сцена локации с боевой системой |
| `TrainingScene` | Сцена тренировки (Training Ground) |
| `WorldScene` | Сцена карты мира |

### NPC Sprite

```typescript
// NPCSprite = Physics.Arcade.Sprite
// Позиция определяется физикой (body.position)

// Движение через setVelocity
npc.moveTo(targetX, targetY, speed);

// Синхронизация данных
npc.x = sprite.x;  // Читаем из физики → записываем в данные
npc.y = sprite.y;
```

## 🎯 Выводы по архитектуре

### Сильные стороны

| Аспект | Решение | Результат |
|--------|---------|-----------|
| **Лаг синхронизации** | Система тиков | Сервер опрашивается только при действиях |
| **Бездействие** | PAUSE_ACTIONS | Нет лишних запросов |
| **Sandbox** | Server-only Storage | Работает без localStorage |
| **Данные** | TruthSystem | Единый источник истины |
| **Бой** | Event Bus | Асинхронная обработка, мгновенный отклик UI |

### Что НЕ нужно менять

| Компонент | Причина |
|-----------|---------|
| **Phaser Physics** | Коллизии работают, хитбоксы настроены |
| **Event Bus** | Асинхронная модель соответствует архитектуре |
| **Tick System** | Решает проблему лагов |
| **TruthSystem** | Память первична — правильный подход |

### Что можно улучшить (опционально)

| Улучшение | Описание | Приоритет |
|-----------|----------|-----------|
| **Batch-синхронизация** | Пачка тиков вместо одиночных | Низкий |
| **Optimistic Update** | Предсказание на клиенте | Низкий |
| **Кнопка "Пробел"** | Промотка времени | Средний |
| **Spatial Hash** | Оптимизация при 100+ объектов | Низкий |

## 📊 Сравнение с альтернативами

### Phaser Render-Only — НЕ НУЖЕН

| Критерий | Текущая архитектура | Render-Only |
|----------|---------------------|-------------|
| **Физика** | ✅ Встроенная | ❌ Своя |
| **Коллизии** | ✅ Автоматически | ❌ Вручную |
| **Лаг** | ✅ Решён тиками | ✅ Решён тиками |
| **Трудозатраты** | ✅ 0 дней | ❌ 1-2 недели |

**Вывод:** Render-Only не даёт преимуществ, т.к. система тиков уже решает проблему лагов.
