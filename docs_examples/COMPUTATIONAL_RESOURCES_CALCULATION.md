## 🧮 Расчёт памяти

### Структура TileData (несжатая)

```
TileData:
  - x, y: int (2 × 4 bytes = 8 bytes)
  - z: sbyte (1 byte)
  - terrain: byte (1 byte)
  - moveCost: float (4 bytes)
  - qiDensity: int (4 bytes)
  - temperature: float (4 bytes)
  - waterDepth: float (4 bytes)
  - objectRef: int (4 bytes, может быть null)
  - flags: byte (1 byte)
  
Итого: ~31 байт на тайл (несжатый)
```

### Расчёт для разных типов локаций

| Тип локации | Тайлов | Несжатый | Сжатый RLE | Sparse (10% заполн.) |
|-------------|--------|----------|------------|----------------------|
| Хутор | 2,500 | 77 KB | ~10 KB | ~8 KB |
| Деревня | 22,500 | 697 KB | ~80 KB | ~70 KB |
| Посёлок | 62,500 | 1.9 MB | ~200 KB | ~195 KB |
| Средний город | 250,000 | 7.7 MB | ~1.5 MB | ~800 KB |
| Большой город | 2,250,000 | 69 MB | ~15 MB | ~7 MB |
| Мегаполис | 25,000,000 | 775 MB | ~150 MB | ~77 MB |
| Храм | 10,000 | 310 KB | ~40 KB | ~31 KB |
| Подземелье | 100,000 | 3.1 MB | ~500 KB | ~310 KB |

### Методы сжатия

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    МЕТОДЫ СЖАТИЯ ТАЙЛОВОЙ СЕТКИ                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   1. RLE (Run-Length Encoding)                                              │
│   ────────────────────────────────────────────────────────────────────────  │
│   Для больших областей одинаковых тайлов:                                   │
│   - Лес: [forest × 1000] → 1 запись вместо 1000                            │
│   - Равнина: [plains × 5000] → 1 запись                                    │
│   - Коэффициент сжатия: до 10-20× для дикой местности                      │
│                                                                              │
│   2. Sparse Array (Разрежённый массив)                                     │
│   ────────────────────────────────────────────────────────────────────────  │
│   Хранить только непустые тайлы:                                           │
│   - Dictionary<Vector2Int, TileData>                                       │
│   - Коэффициент: зависит от заполненности (10-30% типично)                │
│                                                                              │
│   3. Чанковая загрузка (Chunked Loading)                                   │
│   ────────────────────────────────────────────────────────────────────────  │
│   Разбить локацию на чанки 100×100 тайлов (200×200 м):                     │
│   - Загружать только чанки в радиусе видимости                             │
│   - Выгружать далёкие чанки                                                │
│   - Память: O(видимость²) вместо O(локация²)                              │
│                                                                              │
│   4. LOD для тайлов (Level of Detail)                                      │
│   ────────────────────────────────────────────────────────────────────────  │
│   - Близкие тайлы: полные данные                                           │
│   - Средние тайлы: только тип и проходимость                              │
│   - Дальние тайлы: только тип для рендера                                  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

## ⚡ Расчёт производительности

### CPU: Обновление тайлов

| Операция | Сложность | При 1M тайлов |
|----------|-----------|---------------|
| Обновление всех | O(n) | ~10 ms |
| Обновление изменённых (1%) | O(n) | ~0.1 ms |
| Pathfinding (A*) | O(n log n) | ~5-50 ms |
| Обновление Ци | O(n) | ~5 ms |
| Collision check | O(1) per entity | ~0.01 ms |

### GPU: Рендеринг

| Параметр | Значение |
|----------|----------|
| Спрайты на экран | ~1000-5000 |
| Draw calls | ~50-200 |
| Видеопамять | ~100-500 MB |

### 2. Потоковая загрузка

- Фоновый поток загружает чанки заранее
- Приоритет по расстоянию и направлению движения
- Кэширование последних посещённых чанков

### 3. Упрощение данных

| Расстояние от игрока | Детализация |
|---------------------|-------------|
| 0-50 м | Полная (все данные) |
| 50-200 м | Средняя (тип, объекты) |
| 200-600 м | Низкая (только тип) |
| >600 м | Не загружено |

## 📈 Итоговая оценка

### Можно ли запустить игру на ПК?

| Сценарий | Ответ | Примечание |
|----------|-------|------------|
| Малые локации (до 1 км) | ✅ Да | Любой современный ПК |
| Средние локации (1-3 км) | ✅ Да | С чанковой загрузкой |
| Большие локации (3-10 км) | ✅ Да | Требует оптимизаций |
| Мегаполис 10×10 км | ⚠️ С ограничениями | Только с чанковой загрузкой |

### Ключевые оптимизации

1. **Чанковая загрузка** — обязательна для локаций >1 км
2. **Сжатие RLE** — для дикой местности
3. **Sparse Array** — для городов с пустыми зонами
4. **LOD** — для дальних тайлов
5. **Потоковая загрузка** — для плавного перехода

## 🎯 Вывод

**Игра РЕАЛЬНО запустить на ПК** при условии:

1. Размер локации — динамический (100 м ... 10 км)
2. Чанковая загрузка для больших локаций
3. Сжатие тайловой сетки
4. Оптимизация рендера (batching, culling)

- Загрузка только 2% локации (радиус 600 м)
